# 可复用歌词分析数据规范

本文档用于其他项目复用歌词学习资料制作流程。适用场景是：用户提供歌词文本或 LRC 文件，项目不使用 Supabase，分析结果直接保存为本地 JSON、Markdown、数据库种子文件或其他本地数据文件。

核心目标：输出的数据必须能直接用于歌词学习页面，且满足当前项目已经确认过的质量要求。

## 一、适用范围

适用：

- 韩语歌词学习资料制作。
- 用户只提供歌词文本，没有音频或封面。
- 本地项目直接读取 JSON 文件。
- 本地项目后续再自行导入数据库。
- 需要逐句翻译、直译、逐词拆解、核心单词和语法说明。

不适用：

- 纯音乐。
- 没有韩文学习价值的纯英文歌词。
- 只做歌词展示，不做学习解析的项目。

如果歌词中既有韩文也有英文，韩文句必须制作完整学习数据；纯英文句可以翻译，但不强制做韩语逐词拆解，也不加入韩语核心单词。

## 二、输入要求

推荐输入：

- 歌曲标题。
- 艺人名，可选。
- 歌词文本，优先使用 LRC。
- 如果是 LRC，保留时间戳。

歌词清洗要求：

- 不使用歌词文件里自带的中文翻译。
- 删除空行、广告、来源标记、翻译者标记。
- 保留真实歌词重复句，不因为重复就删除。
- 如果歌词没有时间戳，仍可制作学习数据，但 `start_time` 和 `end_time` 可以为空。
- 如果歌词有时间戳，必须按播放顺序排列。

## 四、逐句分析要求

每句必须包含：

- 原文。
- 自然意译。
- 韩语词序直译。
- 逐词拆解。
- 文法详解，可为空。
- 难度等级。
- 朗读文本。

质量要求：

- 原文必须和歌词行一致。
- 自然意译要重新翻译，不使用歌词文件自带中文。
- 直译必须按韩语词序解释，不能写成顺滑中文句子。
- 有韩文的句子，逐词拆解不能为空。
- 逐词拆解必须覆盖句子里的主要韩语词，不只挑一两个词。
- 重复歌词可以复用分析思路，但仍要保留对应行的数据。

## 五、直译规范

直译用于帮助学习者看懂韩语结构，不追求中文自然。

合格示例：

```text
저녁=傍晚 / 노을=晚霞 / 하늘을=天空-宾格 / 바라보면=望着时
```

不合格示例：

```text
当我望向映着晚霞的天空
```

原因：这是自然意译，不是直译。

直译要求：

- 按韩语出现顺序写。
- 助词和词尾要体现出来，例如“宾格”“主题”“如果/当……时”。
- 不要强行润色成自然中文。
- 不要省略影响理解的助词和词尾。

## 六、逐词拆解规范

每个词卡必须包含：

- `surface`：歌词中出现的形式。
- `lemma`：原形或词典形。
- `meaning_zh`：符合歌词语境的中文意思。
- `part_of_speech`：词性。
- `level`：只允许 `L1`、`L2`、`L3`、`L4`。
- `pronunciation`：罗马音或清晰可读发音。

等级规则：

- `L1`：高频基础词、基础助词、基础表达。
- `L2`：常用动词、形容词、连接词尾和中等难度表达。
- `L3`：较抽象、较书面或歌词中不那么基础的表达。
- `L4`：低频、复杂、惯用或需要较高理解门槛的表达。

禁止：

- 不要混用 `A1/A2/B1/B2`。
- 不要缺少发音。
- 不要把英文词、音效词当作韩语核心词。
- 不要把同一句的逐词拆解留空。

## 七、核心单词规范

核心单词是整首歌值得学习者重点记住的词，不是所有词的简单去重。

每条核心单词必须包含：

- 歌词中出现的形式。
- 原形。
- 中文意思。
- 词性。
- 难度等级。
- 发音。
- 出现次数。
- 来源行号。
- 优先级。
- 朗读文本。

要求：

- 必须来自真实歌词。
- 来源行号必须能对应歌词行。
- 词义必须符合该歌语境。
- 高频、常用、能帮助理解歌曲主题的词优先。
- 不要加入歌词里没有的“补充词”。

## 八、文法详解规范

文法详解的原则：**言之有物，有则写，无则空。**

文法详解只回答一个问题：这句韩语有什么具体语法、句型、助词、词尾、缩略或惯用表达值得说明？

允许为空：

- 没有明确语法点。
- 只是普通名词短语。
- 只是英文句。
- 只是重复感叹词或音效词。

必须填写：

- 出现明确语法结构，例如 `-(으)면`、`-고`、`-아/어도`、`-ㄹ 수 있다/없다`、`처럼`、`까지`、`대로`、`게 되다`。
- 出现值得说明的助词组合。
- 出现口语缩略或惯用表达。
- 直译看不出结构，需要解释才能理解。

禁止写：

- “这句适合按整句理解。”
- “这句适合按整句记忆。”
- “先抓核心词，再结合语境理解。”
- “这里表达了歌曲情绪。”
- “没有明显语法点。”
- “没有复杂语法。”

这些都是空话。没有语法点时，字段留空。

容易误判：

- `고고베베`、`아이고` 里的 `고` 不是连接词尾 `-고`。
- `최면` 不是 `-(으)면`。
- `내게`、`네게`、`너에게`、`모든 게` 不应简单写成 `-게` 语法说明。
- `무인도` 里的 `도` 不是“也”的助词。

如果无法确认是不是语法点，留空，不写猜测。

## 九、制作流程

推荐顺序：

1. 清洗歌词。
2. 按行建立 `lyric_lines`。
3. 给每行制作自然意译。
4. 给每行制作韩语词序直译。
5. 给每个韩文句子制作逐词拆解。
6. 判断是否有真实语法点，有则写文法详解，无则留空。
7. 汇总核心单词。
8. 检查所有等级、发音、行号和空字段。
9. 输出本地 JSON 文件。
10. 用质检清单复查。

## 十、质检清单

输出前必须确认：

- 歌词行没有空原文。
- 行号从 1 开始，顺序连续。
- 每条分析都能对应一条歌词行。
- 有韩文的句子都有逐词拆解。
- 逐词拆解没有缺字段。
- 核心单词没有缺字段。
- 所有等级都是 `L1-L4`。
- 核心单词来源行号有效。
- 自然意译不是旧歌词翻译。
- 直译是韩语词序，不是自然中文。
- 文法详解没有空话套话。
- 文法详解没有明显误判。

合格标准：

- 空歌词行：0
- 韩文句子缺逐词拆解：0
- 逐词拆解缺字段：0
- 核心单词缺字段：0
- 非 `L1-L4` 等级：0
- 核心单词来源行号无效：0
- 文法详解空话套话：0
- 文法详解明显误判：0

## 十一、本地项目注意事项

其他项目不用 Supabase 不影响这套分析标准。

区别只在数据保存方式：

- 当前网站把数据写入线上内容库。
- 其他项目可以把同样结构保存成 JSON 文件。
- 如果项目有自己的本地数据库，可以把 JSON 再转换成数据库记录。

不变的是质量要求：

- 每句要能独立显示学习资料。
- 每个韩文句子要有逐词拆解。
- 核心单词要能追溯到歌词行。
- 文法详解不能用套话凑数。
- 没有语法点时，文法详解必须允许为空。

```text
3. 有韩文的句子必须提供逐词拆解。

```
