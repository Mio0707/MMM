# 歌曲数据上线与质检标准流程

本文档用于规范新增歌曲、制作歌词学习解析、上线后质检和问题修复流程。目标是避免再次出现：歌曲已上线但学习数据不显示、核心单词缺失、逐词拆解为空、等级格式不一致、空歌词行、旧翻译混入、语法说明空话套话等问题。

如果要在其他本地项目中复用歌词分析标准，不使用线上内容库或 Supabase，请使用 `docs/reusable-lyric-analysis-standard.md`。那份文档保留通用质量标准，去掉了当前网站专属的上线和数据库步骤。

## 一、上线前文件检查

每首歌至少需要：

- 音频文件：`.mp3`
- 歌词文件：`.lrc`
- 封面文件：可选；没有封面时允许先留空，但必须在记录中明确

检查要求：

- 音频文件能读取，文件大小明显大于 0。
- 歌词文件必须有时间戳，格式如 `[00:12.34]歌词`。
- 歌词必须能按 UTF-8 正常读出韩文；如果终端显示乱码，需要用脚本确认实际读取结果。
- 歌词中附带的旧中文翻译不得写入线上数据。
- 空时间戳行不得写入 `lyric_lines`。
- 文件名不一致时必须显式建立映射，例如 `High Tension.lrc` 对应 `춤을 춰 (High Tension).mp3`。
- 已上线重复歌曲不得重复添加。
- 纯音乐、卡拉 OK 版本、纯英文版本需要单独标注；没有韩文学习价值时不制作韩语学习解析。

## 二、基础歌曲数据导入

基础数据包括：

- `songs`
- `lyric_lines`
- 音频存储路径
- 封面存储路径，可为空

导入规则：

- 先运行检查模式，确认歌曲 ID、歌曲名、歌词行数、韩文行数、是否重复。
- 检查通过后再正式写入。
- `songs.lyrics` 只能保存清洗后的 LRC，不保留旧中文翻译。
- `lyric_lines.original_ko` 不允许为空。
- `lyric_lines.line_index` 必须从 1 开始按播放顺序递增。
- `start_time` 必须有效；`end_time` 应等于下一行开始时间，最后一行可为空。
- 写入失败时必须回滚已写入的歌曲、歌词行和已上传文件。

## 三、学习解析制作规范

每首有韩文歌词的歌曲，必须制作：

- 逐句解析：`lyric_line_analysis`
- 核心单词：`lyric_words`
- 每句内部逐词拆解：`word_breakdown_json`

### 逐句解析字段要求

每条逐句解析必须包含：

- `song_id`
- `lyric_line_id`
- `line_index`
- `original_ko`
- `natural_translation_zh`
- `literal_translation_zh`
- `word_breakdown_json`
- `grammar_note_zh`
- `level`
- `priority_score`
- `tts_text`

质量要求：

- `original_ko` 必须和对应歌词行一致。
- `natural_translation_zh` 重新翻译，不使用歌词文件自带翻译。
- `literal_translation_zh` 必须按韩语词序解释，不写成自然中文句子。
- 韩文句子的 `word_breakdown_json` 不允许为空。
- 纯英文句可以不做韩语逐词拆解，但不应混入核心韩语词库。

### 文法详解规范

`grammar_note_zh` 的原则是：**言之有物，有则写，无则空。**

文法详解只解决“这句韩语为什么这样说”的问题，不承担翻译、赏析、总结情绪或补页面完整度的作用。没有明确语法点时，留空就是合格数据。

允许为空：

- 如果该句没有值得学习者注意的韩语语法、句型、连接词尾、助词用法、敬语、缩略、惯用表达或口语结构，`grammar_note_zh` 应留空。
- 不得为了“看起来完整”而硬写语法说明。

必须填写的情况：

- 出现明确语法结构，例如 `-(으)면`、`-고`、`-아/어도`、`-ㄹ 수 있다/없다`、`처럼`、`까지`、`대로`、`게 되다` 等。
- 出现值得说明的助词或词尾组合。
- 出现口语缩略、惯用表达、语气表达。
- 直译难以看出句子结构，需要通过语法说明帮助理解。

禁止内容：

- 禁止空话套话，例如：
  - “这句适合按整句理解。”
  - “这句适合按整句记忆。”
  - “这句适合整体记忆。”
  - “先抓核心词，再结合语境理解。”
  - “先抓住核心表达。”
  - “这里表达了歌曲情绪。”
  - “没有明显语法点。”
  - “没有复杂语法。”
- 禁止把意译、情绪赏析、歌词理解写进文法详解。
- 禁止为了避免空字段而批量填充模板句。

容易误判的情况：

- 不要只因为词里含有某个字形就判定为语法点。例如 `고고베베`、`아이고` 里的 `고` 不是连接词尾 `-고`。
- 不要把固定词或完整词误拆成语法结构。例如 `최면` 不是 `-(으)면`，`내게`、`네게`、`너에게`、`모든 게` 不应简单写成 `-게` 语法说明。
- 不要把词内部音节当作助词。例如 `무인도` 里的 `도` 不是“也”的助词。
- 如果无法确认是否是真实语法点，宁可留空，不写猜测性说明。

质检规则：

- `grammar_note_zh` 可以为空。
- 如果 `grammar_note_zh` 非空，必须能指出具体语法点或语言现象。
- 数据修复时，不允许用通用模板补齐空语法字段。
- 旧数据中已有的空话套话，应清空或改写成真实语法说明。
- 修复内容数据时，只改数据库中的内容字段，不通过前端隐藏、过滤或兜底显示来掩盖内容质量问题。

## 四、逐词拆解字段要求

`word_breakdown_json` 中每个词卡必须包含：

- `surface`
- `lemma`
- `meaning_zh`
- `part_of_speech`
- `level`
- `pronunciation`

要求：

- 不允许空字段。
- `level` 必须统一使用 `L1`、`L2`、`L3`、`L4`。
- 禁止混用 `A1`、`A2`、`B1`、`B2`。
- 如果源数据来自 A/B 等级，统一映射：
  - `A1/A2` -> `L1`
  - `B1` -> `L2`
  - `B2` -> `L3`
  - `C1/C2` -> `L4`
- `pronunciation` 必须有罗马音或可读发音。
- 英文单词、纯音效词不作为韩语核心词。

## 五、核心单词字段要求

`lyric_words` 每条必须包含：

- `song_id`
- `surface_form`
- `lemma`
- `meaning_zh`
- `pronunciation`
- `part_of_speech`
- `level`
- `frequency_in_song`
- `source_line_indexes`
- `priority_score`
- `tts_text`

要求：

- `level` 必须是 `L1-L4`。
- `source_line_indexes` 必须能对应真实歌词行。
- 核心单词应来自该歌曲歌词，不添加无来源词。
- 词义应符合歌词语境。

## 六、导入前强制检查

正式写入学习解析前，必须检查：

- 每条解析都能匹配到线上 `lyric_lines`。
- 每条韩文解析都有非空 `word_breakdown_json`。
- 每个词卡字段完整。
- 所有等级都是 `L1-L4`。
- `words.source_line_indexes` 能映射到对应歌词行。
- `grammar_note_zh` 为空时允许通过。
- `grammar_note_zh` 非空时，必须包含具体语法点或语言现象。
- 目标歌曲还没有旧的 `lyric_line_analysis` 和 `lyric_words`，除非本次任务明确是更新或修复。

检查失败时不得写入。

## 七、上线后质检

每批上线后必须做全量或目标歌曲质检。

### 必查项

- 歌曲是否存在。
- 音频地址是否存在。
- 封面是否存在；允许为空时要有记录。
- `songs.lyrics` 是否仍包含外部翻译标记，如“翻译”“翻譯”等。
- `lyric_lines` 行数是否合理。
- 是否存在空歌词行。
- 是否存在有韩文歌词但没有逐句解析的歌曲。
- 是否存在有韩文歌词但没有核心单词的歌曲。
- 是否存在韩文逐句解析但逐词拆解为空。
- 是否存在逐句解析关键字段为空。
- 是否存在逐词拆解字段为空。
- 是否存在等级不是 `L1-L4`。
- 是否存在核心单词来源行号无效。
- 是否存在无具体语法点的文法详解套话。
- 是否存在明显误判的文法详解，例如把词内部音节误认为连接词尾或助词。

### 合格标准

上线后目标歌曲必须满足：

- 空歌词行：0
- 韩文句子缺逐词拆解：0
- 逐句解析关键字段为空：0；`grammar_note_zh` 除外，允许为空
- 逐词拆解关键字段为空：0
- 非 `L1-L4` 等级：0
- 核心单词缺失：0
- 核心单词来源行号失效：0
- 文法详解空话套话：0
- 文法详解明显误判：0

## 八、常见问题与处理方式

### 页面不显示学习数据

优先检查：

- 当前歌曲 `songId` 是否正确。
- 匿名权限是否能读取 `lyric_line_analysis` 和 `lyric_words`。
- `level` 是否为页面支持的 `L1-L4`。
- 默认等级页是否有数据。

### 有核心单词，但句子里没有逐词拆解

原因通常是生成解析时只写了 `lyric_words`，没有写每句的 `word_breakdown_json`。

处理：

- 补齐每条韩文句子的 `word_breakdown_json`。
- 导入脚本必须阻止空逐词拆解写入。

### 文法详解为空

处理：

- 先判断是否真的有语法点。
- 有真实语法点：补具体说明。
- 没有真实语法点：保持为空，不修。

### 文法详解是套话

处理：

- 如果能识别真实语法点，改写成具体说明。
- 如果没有真实语法点，清空。
- 不允许批量填充模板句。
- 修复后必须重新检查，确认空话套话和明显误判数量都为 0。

### 文法详解误判了语法点

处理：

- 先看该表达是不是一个完整词、固定表达或人名歌名的一部分。
- 如果只是词内部音节相同，不写语法说明，直接清空。
- 如果确实是语法结构，再补充具体作用，例如“表示条件”“表示并列连接”“表示让步”等。
- 不要因为某个字形看起来像助词或词尾，就自动生成文法详解。

### 空歌词行

处理：

- 删除空 `lyric_lines`。
- 删除对应的空 `lyric_line_analysis`。
- 不要删除正常歌词行，也不要重排行号，除非确认前端依赖已处理。

### 歌词字段和拆分歌词行数不一致

处理：

- 先确认是否因为空时间戳行、重复行、旧歌词残留造成。
- 不直接覆盖时间戳数据。
- 必要时用原始 LRC 重新生成 `lyric_lines`，并同步修正解析引用。

## 九、推荐执行顺序

1. 检查文件格式和重复歌曲。
2. 导入基础歌曲数据。
3. 验证歌曲、音频、歌词行。
4. 制作学习解析。
5. 导入前检查解析完整性。
6. 正式写入解析和核心单词。
7. 上线后质检。
8. 修复质检问题。
9. 再次质检，确认问题为 0。
10. 记录本次发现的新问题类型，并更新本流程文档，避免同类问题重复出现。

## 十、必须避免的问题清单

- 不要把歌词文件里的旧中文翻译写入线上。
- 不要写入空歌词行。
- 不要让韩文句子的逐词拆解为空。
- 不要混用 `A1/A2/B1/B2` 和 `L1-L4`。
- 不要让 `word_breakdown_json` 缺 `pronunciation`。
- 不要只导入核心单词而不导入句内逐词拆解。
- 不要重复添加已上线歌曲。
- 不要在失败后留下半成品数据。
- 不要用未校验的行号建立核心单词来源。
- 不要为了填满 `grammar_note_zh` 写空话套话。
- 不要把词内部音节误判成语法点。
- 不要通过修改前端显示逻辑来绕过内容数据问题。
